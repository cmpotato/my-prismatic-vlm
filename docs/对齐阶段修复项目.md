# Align Stage Repair Project

更新时间：2026-03-19

## 1. 项目目标

在当前 `feature/multi-image` 分支上，修复 `align` 阶段与多图交错前向逻辑之间的两个关键不一致点，使
`DINOv3 + Qwen3VL-text` 的 projector-only 对齐训练重新满足当前代码契约：

1. `align` 数据必须在 tokenization 后保留真实的 `<image>` placeholder。
2. `align` 标签构造不能再假设 tokenizer 自动提供 BOS。

本次修复只针对 `align` 阶段最小闭环，不扩展到新的 LLM backbone 接入，也不改动多图 `finetune` 逻辑。

## 2. 背景问题

### 2.1 风险点一：AlignDataset 与当前 forward 契约不一致

当前分支中的 `PrismaticVLM.forward()` 已改为：

- 从 tokenized 后的 `input_ids` 中查找 `<image>` token
- 用对应图像的 projected patch block 替换该 placeholder
- 要求“图像数量 == placeholder 数量”

但旧版 `AlignDataset` 实际只构造了 `caption + eos`，并未把 `<image>` token 放入 `input_ids`。
这会导致 `align` 阶段的多模态样本在进入当前前向路径时缺少定位锚点。

### 2.2 风险点二：标签构造沿用了错误的 BOS 假设

旧版 `AlignDataset` 会无条件执行：

```python
labels[0] = IGNORE_INDEX
```

这建立在“第 0 个 token 是 BOS，图像 patch 会被插入到 BOS 后面”的旧假设上。
但 Qwen3VL tokenizer 路线并不满足这个假设。继续沿用该逻辑会让首个真实文本 token 静默失去监督。

## 3. 本次修复内容

修改文件：

- `prismatic/preprocessing/datasets/datasets.py`

### 3.1 AlignDataset 现在显式保留 `<image>` placeholder

旧逻辑：

```python
self.prompt_template = "{caption}" + self.tokenizer.eos_token
```

新逻辑：

```python
self.prompt_template = f"{self.image_token}\n{{caption}}{self.tokenizer.eos_token}"
```

这样 `align` 样本在 tokenization 后就会含有一个真实的 `<image>` token，
从而与当前 `PrismaticVLM.forward()` 的 placeholder-replacement 路径对齐。

### 3.2 AlignDataset 启动时校验 `<image>` 是否是单个 token

新增 `_resolve_image_token_id()`：

- 使用 `tokenizer("<image>", add_special_tokens=False)` 获取 token ids
- 强制要求结果长度为 1
- 如果 tokenizer 不能把 `<image>` 保留为单个 token，则直接报错

这样可以避免训练静默跑偏，并把“tokenizer 不支持当前多模态定位契约”的问题提前暴露。

### 3.3 label mask 改为按 placeholder token 处理

旧逻辑：

```python
labels = copy.deepcopy(input_ids)
labels[0] = IGNORE_INDEX
```

新逻辑：

```python
labels = input_ids.clone()
labels[input_ids == self.image_token_id] = IGNORE_INDEX
```

新规则不再依赖 BOS 是否存在，而是只屏蔽 `<image>` placeholder 本身。
后续在 `PrismaticVLM.forward()` 中，这个 placeholder 会被 patch block 替换，
对应 patch block 的 label 也会继续被设为 `IGNORE_INDEX`，因此训练目标仍然只落在 caption 文本上。

## 4. 修复后的 align 阶段语义

单图样本现在的目标序列形态是：

```text
<image>
{caption}</eos>
```

训练时：

- `<image>` token 只作为图像插入锚点，不参与监督
- 视觉 encoder 输出经 projector 投影后，替换 `<image>` 对应位置
- caption token 继续作为 align 阶段的监督目标

这与当前多图交错实现的核心契约一致，只是 `align` 阶段仍然只使用单图单 placeholder。

## 4.1 修改前的 align 训练方法

为了理解这次修复，先明确“修改前”仓库实际上想做什么。

`align` 阶段的总体目标一直都是：

- 冻结 vision backbone
- 冻结 LLM backbone
- 只训练 projector

也就是说，`align` 并不是 instruction tuning，而是把视觉特征映射到 LLM 可消费的嵌入空间中。

旧版 `AlignDataset` 的处理方式延续了早期 LLaVA 风格：

- 原始样本通常是两轮对话
- `human` 轮里带有提示语和 `<image>`
- `gpt` 轮里是图像描述文本

但在旧的 `align` 逻辑里，真正进入监督的并不是整段对话，而是只取：

```python
conversation[-1]["value"]
```

也就是 assistant/gpt 给出的图像描述文本，也可以理解为图文对里的 “caption”。

因此，旧 align 阶段的训练语义本质上是：

- 给定图像
- 让模型生成这张图对应的 caption

而不是：

- 给定图像和一条用户问题
- 让模型生成对话式回答

旧版前向实现中，视觉 patch block 不是按 `<image>` placeholder 位置插入的，而是直接拼在第一个文本 token 后面。
这也是为什么旧数据侧会写死：

```python
labels[0] = IGNORE_INDEX
```

它默认第 0 个位置承担的是类似 BOS / 前缀锚点的角色，而不是正常文本监督位置。

所以，修改前的 align 阶段可以概括成一句话：

> 用冻结的 vision encoder 和冻结的 LLM，训练 projector，让“图像前缀”能够驱动 LLM 生成 caption。

## 4.2 修复后每个张量是怎么流动的

这次修复并没有改变 align 的优化目标，只是把数据和前向对齐到当前多图分支的实现契约。

### 数据集侧

`AlignDataset` 现在会把每条样本构造成：

```text
<image>
{caption}</eos>
```

tokenize 之后得到：

- `input_ids`: 一维文本 token 序列，包含一个真实的 `<image>` token
- `labels`: 初始复制自 `input_ids`，然后把 `<image>` 对应位置设为 `IGNORE_INDEX`
- `pixel_values`: 单图样本统一整理成 `[1, C, H, W]` 或 `[1, ...]`

这里有两个关键点：

1. `<image>` 现在被显式保留在文本里，作为图像插入锚点。
2. 单图样本显式带上 leading dimension `1`，表示“这一条样本有 1 张图”。

### Collator 侧

批处理之后，collator 会统一生成：

- `input_ids`: `[B, T_pad]`
- `labels`: `[B, T_pad]`
- `attention_mask`: `[B, T_pad]`
- `pixel_values`: `[B, max_n_images, C, H, W]` 或对应 dict 版本
- `image_attention_mask`: `[B, max_n_images]`
- `multimodal_indices`: 当前 batch 里哪些样本是多模态

对 align 单图任务来说，通常 `max_n_images = 1`，所以 `pixel_values` 的形状会变成 `[B, 1, C, H, W]`。

### Forward 侧

进入 `PrismaticVLM.forward()` 后，整体流程如下：

1. 先用 `input_ids` 从 LLM 取到纯文本 embedding：
   - `input_embeddings`: `[B, T_pad, D]`
2. 根据 `image_attention_mask` 把批内真实存在的图像展平：
   - align 单图场景下，通常变成 `[B, C, H, W]`
3. 图像经过 vision backbone：
   - `patch_features`: `[n_total_images, n_patches, vision_dim]`
4. patch features 经过 projector：
   - `projected_patch_embeddings`: `[n_total_images, n_patches, llm_dim]`
5. 在每条样本的 `input_ids` 中查找 `<image>` token 位置
6. 用对应图像的 patch block 替换该位置

替换后的序列不再包含 `<image>` 这个文本 token，而是变成：

- placeholder 前的文本 embedding
- 图像对应的 patch block
- placeholder 后的文本 embedding

与此同时：

- patch block 对应的 attention mask 全为有效位置
- patch block 对应的 labels 全部设为 `IGNORE_INDEX`
- caption 文本部分继续保留真实 labels

最后，所有样本重新 pad 成：

- `fused_embeddings`
- `fused_attention_mask`
- `fused_labels`

再直接用 `inputs_embeds=fused_embeddings` 喂给 LLM。

因此，最终真正参与 loss 的只有 caption token；图像占位和图像 patch block 都不参与 loss。

## 4.3 为什么修复后仍然保持“原始对齐语义”

这部分可以简要归结为三个不变项。

### 监督文本没有变

修复后，align 数据仍然只取：

```python
conversation[-1]["value"]
```

也就是 assistant/gpt 的 caption。human 那一轮问题文本依然没有进入监督目标。

所以优化目标依然是：

- 输入：图像 + 一个视觉占位
- 预测：caption 文本

而不是 instruction tuning 中常见的：

- 输入：用户问题 / 多轮对话
- 预测：回答文本

### loss 的落点没有变

修复后：

- `<image>` placeholder 本身是 `IGNORE_INDEX`
- 替换进去的 patch block 仍然全部是 `IGNORE_INDEX`
- 只有 caption 文本 token 参与 loss

因此训练目标仍然是：

```text
maximize log p(caption tokens | visual prefix)
```

也就是原始 align 阶段的“图像到 caption 对齐”目标。

### 可训练参数集合没有变

`stage=align` 时依然是：

- 冻结 vision backbone
- 冻结 LLM backbone
- 只训练 projector

所以模型学到的仍然只是“如何把 DINOv3 视觉特征映射到 Qwen 文本空间”，而不是去更新 LLM 的指令跟随能力。

换句话说，这次修复改变的是视觉信息注入的实现方式：

- 从“默认插在第一个 token 后”
- 改成“显式替换 `<image>` placeholder”

但没有改变 align 阶段的监督对象、loss 位置和可训练参数集合，因此它仍然保持了原始对齐语义。

## 5. 影响范围

本次修改仅影响：

- `align` 阶段样本的文本构造
- `align` 阶段样本的 label mask 方式

本次修改不影响：

- `finetune` / `full-finetune` 数据集结构
- 多图 `images` 字段处理
- `PrismaticVLM.forward()` 的多图交错替换逻辑
- LoRA、checkpoint、训练策略等训练基础设施

## 6. 验证建议

建议按以下顺序做验证：

1. 语法级验证
- 对改动文件执行 `py_compile`

2. 数据级 smoke test
- materialize 一个 `align` 样本
- 检查 `input_ids` 中恰好存在一个 `<image>` token
- 检查对应位置 label 为 `IGNORE_INDEX`

3. 前向级 smoke test
- 跑单 batch `align` forward
- 确认不再出现“image 数量与 placeholder 数量不一致”错误

4. 训练级 smoke test
- 用极小 `align_max_steps` 启动 1 卡或 8 卡短跑
- 确认 loss 为有限值且 checkpoint 正常保存

## 7. 剩余风险

1. 当前修复默认 tokenizer 已支持 `<image>` 作为独立 token
- 对 Qwen3VL-text 路线这是当前分支已经建立的契约
- 对其他旧 LLM backbone，若未注册 `<image>` token，`align` 现在会更早、更明确地失败

2. `align` 阶段仍然保留“丢弃 human prompt，只监督 caption”的旧语义
- 这符合当前框架和既有对齐数据格式
- 但对 VQA 风格数据而言，仍然不是“图 + 问题 -> 答案”的训练方式

## 8. 本次改动的设计原则

本次修复遵循以下原则：

- 只修补 `align` 阶段与当前前向契约的断裂点
- 不回退多图交错实现
- 不引入新的模型接入或训练策略改造
- 优先保证错误显式暴露，而不是静默兼容

## 9. 后续可选工作

如果后续继续沿当前分支推进，可考虑：

1. 给 `AlignDataset` 增加最小单测或 smoke script。
2. 将 `<image>` placeholder 支持抽象为所有 LLM backbone 的统一契约。
3. 若后续切换到新的 Qwen backbone（如 Qwen3.5 路线），优先先验证 tokenizer 对 `<image>` 的保留行为。
